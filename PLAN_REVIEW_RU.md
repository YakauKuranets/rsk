# Ревью предложенного плана развития (Spider/Fuzzer)

## Краткая оценка

План сильный и зрелый по охвату: есть логика от разведки к верификации, акцент на логирование, модульность и контроль воздействия.
Для перехода к «production-grade» уровню я бы оставил общий вектор, но уточнил порядок внедрения и добавил строгие guardrails для безопасности, законности и воспроизводимости результатов.

## Что в плане уже хорошо

- Чёткая модульность и этапность (10 независимых блоков).
- Правильный фокус на read-only/минимальном воздействии.
- Наличие критериев приёмки по каждому модулю.
- Привязка к практической верификации риска, а не только к детекту.

## Что улучшить в архитектуре перед реализацией

### 1) Ввести обязательные guardrails (до любых новых модулей)

- Режимы запуска: `passive`, `safe-active`, `explicit-verify`.
- Политики области теста: allowlist CIDR/host/port и denylist критичных узлов.
- Kill-switch и глобальные лимиты: RPS, concurrency, budget по времени.
- Доказуемый non-destructive контракт: запрет write-операций по умолчанию.

### 2) Унифицировать движок заданий

- Единый `JobRunner` с очередью, retry-политиками, дедлайнами и отменой.
- Нормализованный формат событий: `task_id`, `target_id`, `module`, `action`, `result`, `evidence_ref`.
- Переиспользуемый `EvidenceStore` (локально) для stdout/pcap/http transcript.

### 3) Сначала стабилизировать data model

- Отдельные сущности: `Target`, `Credential`, `Finding`, `Evidence`, `Campaign`.
- Явные статусы проверки уязвимостей: `suspected`, `verified-safe`, `not-reproducible`.
- Версионирование схемы БД и миграции.

## Рекомендованный порядок внедрения (изменённый)

### Wave A — «безопасный фундамент» (1–2 спринта)

1. Adaptive Credential Auditor (в safe-rate режиме).
2. Session Security Checker.
3. Smart API Fuzzer (только soft payloads + строгие таймауты).

Почему: высокая ценность, минимальный операционный риск, быстрый ROI.

### Wave B — «углублённая верификация»

4. RCE PoC verifier (только безвредные read-only команды, deny write).
5. Breach Data Enrichment (с офлайн режимом и анонимизацией).
6. Feedback Learning (локальная база приоритетов).

Почему: усиливает точность отчётов и ускоряет последующие кампании.

### Wave C — «повышенный риск/сложность»

7. Lateral Movement Scanner.
8. Passive Traffic Analyzer.
9. Persistence Assessment.
10. Exploit Auto-Integration.

Почему: выше юридические/операционные риски и требования к изоляции.

## Комментарии по исходным 10 пунктам

- **1. Adaptive Credential Auditor** — отличный приоритет; добавить «этический budget»: не более N попыток/учётку/час.
- **2. RCE PoC** — нужен список только безопасных команд по ОС + запрет shell redirection и pipe.
- **3. Lateral Movement** — включать только при флаге `allow_pivot=true` и отдельной авторизации.
- **4. Passive Traffic** — предусмотреть маскирование PII и ротацию pcap.
- **5. Session Checker** — добавить проверку Secure/HttpOnly/SameSite и TTL сессии.
- **6. Smart API Fuzzer** — сначала «schema-aware fuzzing», затем агрессивные мутации.
- **7. Breach Enrichment** — предпочтительно k-anonymity и локальный кэш.
- **8. Persistence Assessment** — лучше перенести в конец, включать только на лабораторных стендах.
- **9. Exploit Auto-Integration** — только в sandbox/VM с read-only сети и ручным подтверждением.
- **10. Feedback Learning** — полезно рано, но хранить знания в explainable виде (без «чёрного ящика»).

## Дополнительные критерии приёмки (добавить ко всем модулям)

- Повторяемость: одинаковый результат при повторе на одном и том же стенде.
- Безопасность исполнения: 0 необратимых изменений на цели.
- Наблюдаемость: полный аудиторский трейл + экспорт в отчёт.
- Управляемость: runbook для инцидентной остановки модулей.
- Производительность: верхняя граница на время/ресурсы кампании.

## Итог

План очень хороший как стратегический каркас.
Я бы утвердил его, но реализовывал через «safety-first» порядок (Wave A→B→C), чтобы получить быстрый результат без роста операционных и юридических рисков.
