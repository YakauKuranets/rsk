# Аудит проекта Hyperion (быстрый технический обзор)

## Что обнаружено

- Проект — desktop-приложение на **Tauri 2 + React 19 + Vite 7** с Rust backend.
- Фронтенд содержит один очень большой «оркестратор» (`src/App.jsx`) с большим количеством состояний и бизнес-логики.
- В backend присутствуют модули для сканирования, стриминга (FFmpeg/HLS/WS), FTP, аудита и «паука».
- Есть документация архитектуры и отдельный скрипт preflight для Kali Linux.
- В репозитории лежат Windows-библиотеки (`as/*.dll`, `as/*.ocx`) и модель `yolov8s.pt`.
- Обнаружены дублирующиеся файлы в `src-tauri/src/src/*`, похожие на вторую копию backend-части.

## Риски и проблемные зоны

1. **Безопасность и секреты**
   - Во фронтенде есть захардкоженные чувствительные значения (логины/пароли/cookie), что может привести к утечкам.

2. **Поддерживаемость фронтенда**
   - `src/App.jsx` перегружен: UI, состояние, network invoke, orchestration и доменная логика находятся в одном месте.
   - Такой размер усложняет тестирование, ревью и безопасные изменения.

3. **Кросс-платформенность**
   - Одновременно декларируется Linux/Kali-поддержка и хранится Windows-only стек (`as/*.dll/*.ocx`).
   - Нужна явная стратегия feature-gating по платформам.

4. **Сборка и качество**
   - `npm run build` проходит, но есть предупреждения css-minify/syntax и предупреждение о крупном бандле (>500KB).
   - `cargo check` не прошёл в этой среде из-за сетевого доступа к crates.io (403), поэтому валидация Rust части неполная.

5. **Структура репозитория**
   - Присутствуют артефакты/бинарники в корне и потенциально лишние дубликаты исходников.
   - Это увеличивает шум в истории, вес репозитория и риск рассинхронизации.
